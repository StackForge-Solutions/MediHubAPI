## Codex Prompt

```text
Implement the `Pharmacy Transactions` frontend for `Sprint 6` in the pharmacy inventory module using Angular.

Project context:
- This is a pharmacy stock and purchase workflow.
- `Pharmacy Transactions` is Sprint 6 and should present the stock movement ledger generated by purchase receipt, stock adjustment, and future inventory flows.
- The backend is Spring Boot.
- The UI should fit a hospital/pharmacy admin workflow, not an e-commerce style layout.
- Keep the design clean, dense, operationally useful, and audit-friendly.
- Prioritize fast filtering, readable ledger rows, and quick drill-down into source references.

Routes:
- /pharmacy/transactions

Main goals:
- show a full stock movement ledger in descending transaction time order
- allow filtering by medicine, batch, vendor, transaction type, reference, and date range
- open a transaction detail drawer from the list
- allow export of the currently filtered ledger view
- support navigation from a transaction row to the linked stock, purchase order, or adjustment context

Required page sections:

1. Header area
- Page title: Transactions
- Subtitle: Review stock movement history across receipts, adjustments, and other inventory events
- Primary action button: Export

2. Filter bar
- Search by reference number, medicine, note, or batch
- Medicine filter
- Vendor filter
- Transaction type filter
- Batch number filter
- Reference type filter
- From date filter
- To date filter
- Reset filters button

3. Main transaction table
Columns:
- Time
- Medicine
- Batch
- Vendor
- Type
- Qty In
- Qty Out
- Balance After
- Unit Cost
- Unit Price
- Reference
- Created By
- Actions

4. Detail drawer
- Transaction metadata
- Medicine and batch context
- Balance before and after
- Source reference details
- Note
- Quick navigation actions

5. Empty, loading, and error states
- loading placeholders for filter area and table
- empty state when no transactions match filters
- retry option on API errors

6. Shared components and app conventions
- use BannerComponent
- use LoaderBackdropComponent
- use authorization logic(superadmin have all access)
- use HeaderComponent

Design requirements:
- use a compact admin ledger style
- keep numeric columns aligned and easy to scan
- make `qtyIn` and `qtyOut` visually distinct
- use status chips or type chips for transaction types:
  - PURCHASE_RECEIPT
  - SALE
  - ADJUSTMENT_IN
  - ADJUSTMENT_OUT
  - RETURN_TO_VENDOR
  - RETURN_FROM_PATIENT
  - EXPIRED
  - DAMAGED
- keep filters sticky if possible
- preserve filter state in route query params
- desktop-first layout, but remain usable on tablet widths

Navigation behavior:
- opening `/pharmacy/transactions` loads the ledger page
- clicking a row opens a transaction detail drawer
- if `referenceType = PURCHASE_ORDER`, support navigation to /pharmacy/purchase-orders/:id
- if `referenceType = STOCK_ADJUSTMENT`, support navigation to /pharmacy/stock-adjustments/:id
- if `medicineId` exists, support navigation to /pharmacy/stocks/:medicineId
- support prefilled query params such as `medicineId`, `referenceType`, or `referenceId` when navigated from other pages

Use these backend APIs:

GET /api/pharmacy/transactions
Query params:
- q
- medicineId
- vendorId
- transactionType
- batchNo
- referenceType
- referenceId
- fromDate
- toDate
- page
- size

Response shape:
{
  "content": [
    {
      "transactionId": 7001,
      "transactionTime": "2026-03-12T11:00:00Z",
      "medicineId": 101,
      "medicineName": "Paracetamol 500",
      "batchId": 9001,
      "batchNo": "BATCH-APR-01",
      "vendorId": 4,
      "vendorName": "ABC Pharma",
      "transactionType": "PURCHASE_RECEIPT",
      "qtyIn": 100,
      "qtyOut": 0,
      "balanceAfter": 120,
      "unitCost": 1.75,
      "unitPrice": 2.5,
      "referenceType": "PURCHASE_ORDER",
      "referenceId": 501,
      "referenceNo": "PO-20260312-001",
      "createdBy": "admin",
      "note": "Initial receipt"
    }
  ],
  "page": 0,
  "size": 20,
  "totalElements": 1,
  "totalPages": 1,
  "last": true
}

GET /api/pharmacy/transactions/:id
Response shape:
{
  "data": {
    "transactionId": 7001,
    "transactionTime": "2026-03-12T11:00:00Z",
    "transactionType": "PURCHASE_RECEIPT",
    "medicineId": 101,

GET /api/pharmacy/medicines/search
Query params:
- mode
- form
- q
- limit
- inStockOnly

Use for:
- medicine filter autocomplete

GET /api/pharmacy/vendors/options
Query params:
- q
- active

Use for:
- vendor filter dropdown

GET /api/pharmacy/enums
Response shape:
{
  "data": {
    "transactionTypes": [
      "PURCHASE_RECEIPT",
      "SALE",
      "ADJUSTMENT_IN",
      "ADJUSTMENT_OUT",
      "RETURN_TO_VENDOR",
      "RETURN_FROM_PATIENT",
      "EXPIRED",
      "DAMAGED"
    ]
  }
}

Angular implementation requirements:
- create page component: transaction-list.page
- create reusable components if needed:
  - stock-transaction-table
  - stock-filter-bar
  - status-chip
- create a detail drawer component or keep the drawer logic in the page if that matches current app patterns
- create API service for transaction list and detail calls
- keep models typed
- use route query params to preserve filter state
- keep filter form reactive
- support debounced search input
- support server-side pagination

Expected files:
- transaction-list.page.ts
- transaction-list.page.html
- transaction-list.page.scss
- pharmacy-transaction.api.ts
- pharmacy-transaction.model.ts
- any small reusable drawer or table components needed for this flow

Implementation notes:
- do not hardcode data
- do not use client-side filtering for server-driven list data
- format date/time consistently
- format money consistently for unit cost and unit price
- right-align quantity and balance columns
- export should respect the active filter state; if no dedicated export API exists yet, document the chosen frontend limitation clearly
- keep the drawer read-only

Acceptance criteria:
- transaction list loads from API with server-side filters and pagination
- filter state is preserved in query params
- transaction detail drawer loads from API
- row navigation to linked records works when reference metadata exists
- export action is present and aligned with the available API strategy
- loading, empty, and error states are handled
- layout is production-ready and consistent with the pharmacy admin module

Also provide:
- Angular interfaces for request and response models
- a short explanation of component responsibilities
- any assumptions if backend fields differ
```
